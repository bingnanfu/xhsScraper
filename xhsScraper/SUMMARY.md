# 小红书爬虫项目总结

## 项目概述

基于 Playwright 的小红书评论爬取工具，通过搜索关键词 → 点击笔记卡片 → 拦截评论 API → 导出 Excel 的流程，批量采集笔记评论数据。

## 技术架构

- **框架**: Playwright (async) + openpyxl
- **数据获取方式**: 不解析 DOM，而是通过 `page.on("response")` 拦截评论 API 响应（`/api/sns/web/v2/comment/page` 和 `/sub/page`）
- **交互方式**: 先登录保存 Cookie，再搜索关键词自动爬取
- **封装**: 做成了 Claude Code skill（`/xhs`），非技术用户可在对话中直接使用

## 遇到的问题与解决方案

### 问题一：评论爬不完（640 条主评论只爬到 30 条）

**现象**: 页面显示 640 条主评论，实际只爬到 30 条主评论 + 344 条回复。

**根因**: `_do_scroll` 使用 `scrollBy(0, 600)` 固定增量滚动。随着评论加载越来越多，容器高度持续增长，600px 的增量越来越难触及底部，无法触发下一页 API 请求加载。

**解决**:
1. 滚动策略从 `scrollBy(0, 600)` 改为 `scrollTop = scrollHeight`，每次直接跳到容器底部，强制触发懒加载
2. `STALE_LIMIT` 从 6 提高到 12，容忍网络延迟
3. 从 API 响应中读取 `has_more` 字段，`false` 时主动停止，不再盲等超时

**经验**:
- 对于无限滚动/懒加载页面，**滚到底部**比固定增量滚动可靠得多
- 利用 API 响应中的分页元数据（has_more / cursor）做精确的终止判断，比纯靠"连续 N 次无变化"更准确

### 问题二：多篇笔记只能爬第一篇（第二篇始终被判为重复）

**现象**: 爬完第一篇后，点击后续卡片全部被标记为"已爬取过"跳过。

**根因**: 弹窗关闭后 URL 可能仍包含 `/explore/{旧ID}`。下一次点击卡片后，`wait_for_function("url.includes('/explore/')")` 因为 URL 本来就包含 `/explore/` 而立刻通过，拿到的还是旧 ID，触发去重逻辑。

**解决**:
1. 点击前检查：如果 URL 还在 `/explore/`，说明上次弹窗没关干净，先强制关闭
2. 点击后等待：改为等待 URL 变成一个**与点击前不同的** `/explore/` 地址，传入 `url_before_click` 做对比
3. `_close_note_detail` 判断条件加严：必须同时满足 `"search_result" in url` 且 `"/explore/" not in url`
4. 所有关闭方式都尝试后仍未回到搜索页，强制 `page.goto(search_url)` 兜底

**经验**:
- SPA 页面中 URL 状态管理容易出问题，**点击前记录 URL、点击后等待 URL 变化**是通用的可靠模式
- 弹窗/模态框的关闭不能假设一次就成功，需要多种方式级联尝试 + 最终兜底
- 去重逻辑要在确认新页面真正打开后再做，不能在 URL 可能是旧状态时就判断

## 通用经验（适用于类似爬虫项目）

### 滚动加载
- 懒加载容器优先用 `scrollTop = scrollHeight` 而非固定增量
- 找准滚动目标容器（如 `.note-scroller`），不要滚 `window`
- 结合 API 响应的 `has_more` 做终止判断

### API 拦截
- `page.on("response")` 拦截比 DOM 解析更稳定，不受前端渲染变化影响
- 记得用 `seen_ids` 去重，同一条评论可能在多次 API 响应中出现
- 拦截器中用 `nonlocal` 修改外部状态变量

### SPA 页面导航
- 操作前后都要校验 URL 状态
- 等待条件要具体（等"新的 URL"而非"包含某关键词的 URL"）
- 页面状态恢复（关闭弹窗、返回列表页）需要多层 fallback

### Skill 封装
- 不要在 skill 文件中写绝对路径，用相对路径或"相对于本文件所在目录"的描述
- 把脚本和 SKILL.md 放在同一目录下，形成自包含的 skill 包
